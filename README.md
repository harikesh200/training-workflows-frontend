# Maintenance Workflow Console

React frontend for creating and monitoring machine-maintenance workflows.
Operators can upload source files, configure email routing, follow execution
progress, and confirm email delivery through the workflow API.

## Features

- Create workflows from machine logs, an error manual, and a vendor catalog.
- Validate file types, file sizes, SMTP details, and routing addresses before
  submission.
- Maintain an ordered, reorderable list of vendor email addresses.
- Poll active workflows and display their status, progress, and execution
  timeline.
- Show resolved vendor mappings, timestamps, and failure details.
- Report API connectivity and normalize backend, network, and validation
  errors.
- Support light, dark, and system themes.
- Provide keyboard-accessible controls, focus management, status
  announcements, and reduced-motion handling.

## Tech Stack

- React 19 and TypeScript
- Vite 8
- TanStack Router and TanStack Query
- React Hook Form and Zod
- Tailwind CSS 4, shadcn/ui patterns, and Radix primitives
- Axios
- Vitest, React Testing Library, and Mock Service Worker

## Prerequisites

- Node.js
- Corepack or pnpm
- A compatible maintenance workflow API

The repository declares `pnpm@11.5.1` as its package manager.

## Getting Started

1. Install dependencies:

    ```bash
    corepack enable
    pnpm install
    ```

2. Optionally create a local environment file:

    ```bash
    cp .env.example .env.local
    ```

    PowerShell:

    ```powershell
    Copy-Item .env.example .env.local
    ```

3. Start the backend configured by `VITE_API_BASE_URL`.

4. Start the frontend:

    ```bash
    pnpm dev
    ```

Vite serves the application at `http://localhost:5173` by default.

## Environment

| Variable            | Required | Default                 | Description                                                                                       |
| ------------------- | -------- | ----------------------- | ------------------------------------------------------------------------------------------------- |
| `VITE_API_BASE_URL` | No       | `http://localhost:3000` | Base URL for health and workflow requests. Trailing slashes are removed automatically. |

Environment values are validated when the application loads. Because Vite
embeds `VITE_*` values into the client bundle, set production values before
running the build.

## Workflow Inputs

Each workflow submission requires:

| Field             | Requirement                                          |
| ----------------- | ---------------------------------------------------- |
| Machine logs      | CSV file, maximum 10 MiB                             |
| Error manual      | PDF file, maximum 10 MiB                             |
| Vendor catalog    | CSV file, maximum 10 MiB                             |
| SMTP sender email | Valid email address                                  |
| SMTP app password | Non-empty value                                      |
| Vendor emails     | At least one valid email address; order is preserved |
| Plant-head email  | Valid email address                                  |

The form submits these values as `multipart/form-data`. The vendor email list
is serialized as JSON in the `vendorEmails` part.

## Application Routes

| Route                    | Purpose                                           |
| ------------------------ | ------------------------------------------------- |
| `/`                      | Create a workflow                                 |
| `/workflows/$workflowId` | Monitor the workflow created in the current session |

Routes use TanStack Router file-based routing. `src/routeTree.gen.ts` is
generated by the Vite router plugin and must not be edited manually.
`.tanstack/` contains temporary router-generation files and is ignored by Git.

## Backend API Contract

The frontend calls the following endpoints relative to
`VITE_API_BASE_URL`:

| Method | Endpoint                                            | Usage                                                                   |
| ------ | --------------------------------------------------- | ----------------------------------------------------------------------- |
| `GET`  | `/health`                                           | API status indicator; refreshed every 30 seconds                        |
| `POST` | `/v1/workflows`                                     | Create a workflow from multipart form data                              |
| `GET`  | `/v1/workflows/:workflowId`                         | Load workflow details; polled every 1.5 seconds while queued or running |

Successful JSON responses use a `{ "data": ... }` envelope. Backend errors may
use:

```json
{
    "error": {
        "code": "ERROR_CODE",
        "message": "Human-readable message",
        "details": null
    }
}
```

All health, creation, and workflow-detail responses are validated with Zod.
Invalid response shapes are treated as application errors rather than being
rendered as trusted data.

## State and Error Handling

- TanStack Query owns API health and workflow server state.
- Active workflows poll until they succeed or fail.
- Completed and failed workflow status expires according to the backend
  retention window.
- Network and server failures use limited retries; creation requests are never
  retried automatically.
- React Hook Form owns creation-form state and maps backend field errors back
  to their controls.
- The SMTP password is cleared after successful workflow creation.
- Only the selected theme is persisted in browser storage; workflow form
  values and files are not persisted by the frontend.

## Production Deployment

Create the production bundle with:

```bash
pnpm build
```

Deploy the generated `dist/` directory to a static host. Configure:

- `VITE_API_BASE_URL` before building.
- Backend CORS rules for the frontend origin.
- SPA fallback routing to `index.html` so direct workflow-detail URLs resolve.
- HTTPS for both the frontend and API when SMTP credentials are submitted.

`pnpm preview` is intended for local verification, not as a production server.
